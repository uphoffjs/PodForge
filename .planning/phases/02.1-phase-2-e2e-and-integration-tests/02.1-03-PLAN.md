<interfaces>
From src/lib/pod-algorithm.ts:
```typescript
export interface PlayerInfo { id: string; name: string }
export interface RoundHistory { pods: { playerIds: string[]; isBye: boolean }[] }
export interface PodAssignment { pod_number: number; is_bye: boolean; players: { player_id: string; seat_number: number | null }[] }
export interface PodAssignmentResult { assignments: PodAssignment[]; warnings: string[] }

export function generatePods(players: PlayerInfo[], previousRounds: RoundHistory[]): PodAssignmentResult
export function buildOpponentHistory(previousRounds: RoundHistory[]): Map<string, Map<string, number>>
export function buildByeCounts(previousRounds: RoundHistory[]): Map<string, number>
```

The algorithm:
- Accepts active players + previous round history
- Returns pod assignments with seat numbers + warnings
- Throws Error if fewer than 4 active players
- Greedy assignment: builds opponent history matrix, assigns players to pods minimizing repeat opponents
- Bye selection: players with fewest total byes are prioritized, ties broken randomly (Fisher-Yates)
- Seat assignment: random 1-4 for non-bye pods, null for bye pods
- For 5-7 players: produces 1 pod of 4 + bye pod; warns about high bye count for 6-7 players

Existing unit tests in src/lib/pod-algorithm.test.ts already cover individual function behavior.
The integration tests here focus on MULTI-ROUND scenarios testing fairness properties across many rounds.

5. "handles 6 players (high bye count scenario)" —

   - Assert: `max(byes) - min(byes) <= 1`

describe("opponent avoidance across rounds"):
6. "minimizes repeat opponents for 8 players over 4 rounds" —
   - 8 players, 4 rounds
   - After 4 rounds, build opponent history matrix
   - For each pair, the number of times they've been in the same pod should be <= 2 (since with 8 players in 2 pods, there are limited unique pairings)
   - Assert: max repeat count across all pairs is reasonable (no pair meets more than ceil(rounds * 3/7) times, approximately)
   - More practically: assert that the total repeat opponent count is less than what a random assignment would produce. Use a bound: for 8 players in 4 rounds, each player has 3 opponents per round = 12 opponent-slots. With 7 other players, perfectly spread would be ~1.7 each. Assert max pair count <= 3.

7. "minimizes repeat opponents for 12 players over 6 rounds" —
   - 12 players, 6 rounds (3 pods per round)
   - Build opponent history
   - Assert: max pair meet count <= 3 (with 12 players, there are 11 other players, 3 opponents per round, 6 rounds = 18 slots for 11 players = ~1.6 average. Max should be well bounded.)

8. "no repeat opponents in round 2 for 8 players" —
   - 8 players, 2 rounds
   - After round 2, check opponent history
   - For each player, the set of opponents in round 2 should have minimal overlap with round 1
   - Assert: total number of repeat pairs between round 1 and round 2 is 0 (ideal) or <= 2 (acceptable given randomization)

describe("multi-round structural correctness"):
9. "every round has correct pod structure for various player counts" —
   - Test player counts: 4, 5, 8, 9, 12, 13, 16, 17, 20
   - For each, run 3 rounds
   - Assert per round:
     - Total players across all pods = playerCount
     - Non-bye pods have exactly 4 players each
     - Bye pod (if exists) has (playerCount % 4) players
     - Each player appears in exactly one pod per round
     - Seat numbers in non-bye pods are 1-4 (unique within each pod)
     - Seat numbers in bye pods are null

10. "fewer than 4 players throws error" —
    - 3 players, attempt to generate
    - Assert: throws Error with message about fewer than 4 players

11. "all seat numbers are unique within each pod" —
    - 16 players, 5 rounds
    - For each round, for each non-bye pod, assert seat numbers are [1,2,3,4] (as a set)
    - For bye pods, assert all seat numbers are null

12. "seat assignments are randomized (not always 1,2,3,4 in same order)" —

    - Collect all seat assignment orders for pod-1 across rounds

**Important:** Since generatePods uses Math.random internally (Fisher-Yates), these tests verify statistical properties, not exact outputs. Use bounds/inequalities rather than exact equality. For the bye fairness test, `max - min <= 1` is a strong guarantee that the algorithm enforces by design.
