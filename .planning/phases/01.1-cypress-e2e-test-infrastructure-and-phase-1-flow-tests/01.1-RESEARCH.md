# Phase 01.1: Cypress E2E Test Infrastructure and Phase 1 Flow Tests - Research

**Researched:** 2026-02-23
**Domain:** E2E testing with Cypress, visual regression, CI/CD
**Confidence:** HIGH

## Summary

This phase adds Cypress end-to-end test infrastructure and comprehensive tests for all Phase 1 user flows in the Commander Pod Pairer app. The app is a Vite + React 19 + TypeScript + Tailwind CSS v4 frontend backed by Supabase (PostgREST + Realtime). Phase 1 built: event creation via RPC, player join via direct insert, player self-drop via RPC, real-time player list via Supabase Realtime, QR code display, and a mobile-first dark theme with amber/gold accent on purple.

The codebase currently has **zero `data-testid` attributes** on any component, so adding them is a prerequisite for all Cypress specs. There is no existing `.github/workflows/` directory. The project uses `"type": "module"` in package.json, so Cypress config must use ESM-compatible patterns. The Supabase client makes requests to PostgREST endpoints (`/rest/v1/events`, `/rest/v1/players`, `/rest/v1/rpc/create_event`, `/rest/v1/rpc/drop_player`) which are the targets for `cy.intercept()` mocking.

**Primary recommendation:** Use Cypress 14 with `cypress.config.js` (ESM), `eslint-plugin-cypress` v6 with flat config, `cypress-visual-regression` v5 for screenshot comparison at 3 breakpoints, and the `cypress-io/github-action@v6` for CI. All specs in JavaScript. Custom commands for common flows. Mock Supabase PostgREST via `cy.intercept()` for most tests; use real Supabase only for RPC validation flows where RLS/hashing behavior matters.

### Pattern 1: Supabase PostgREST Intercept Mocking

**What:** Mock all Supabase REST API calls via `cy.intercept()` using the PostgREST URL patterns the app actually hits.
**When to use:** All tests except the few that validate real RLS/RPC behavior.

The Supabase JS client translates method calls to PostgREST HTTP requests:

| App Code | HTTP Request | Intercept Pattern |
|----------|-------------|-------------------|
| `supabase.from('events').select(...)` | `GET /rest/v1/events?select=...&id=eq.{uuid}` | `GET **/rest/v1/events*` |
| `supabase.from('players').select(...)` | `GET /rest/v1/players?select=...&event_id=eq.{uuid}` | `GET **/rest/v1/players*` |
| `supabase.from('players').insert(...)` | `POST /rest/v1/players` | `POST **/rest/v1/players*` |
| `supabase.rpc('create_event', {...})` | `POST /rest/v1/rpc/create_event` | `POST **/rest/v1/rpc/create_event` |
| `supabase.rpc('drop_player', {...})` | `POST /rest/v1/rpc/drop_player` | `POST **/rest/v1/rpc/drop_player` |

**Example:**
```javascript
// Source: Cypress official docs + Supabase PostgREST patterns
// Mock event fetch
cy.intercept('GET', '**/rest/v1/events*', {
  statusCode: 200,
  body: [{ id: 'test-uuid', name: 'Friday Night Commander', status: 'active', created_at: '2026-01-01T00:00:00Z' }]
}).as('getEvent')

// Mock player list fetch
cy.intercept('GET', '**/rest/v1/players*', {
  statusCode: 200,
  body: [
    { id: 'p1', event_id: 'test-uuid', name: 'Alice', status: 'active', created_at: '2026-01-01T00:01:00Z' },
    { id: 'p2', event_id: 'test-uuid', name: 'Bob', status: 'active', created_at: '2026-01-01T00:02:00Z' }
  ]
}).as('getPlayers')

// Mock create event RPC
cy.intercept('POST', '**/rest/v1/rpc/create_event', {
  statusCode: 200,
  body: '"new-event-uuid"'
}).as('createEvent')

cy.visit('/')
// ... interact and assert
cy.wait('@createEvent')
```

### Pattern 2: Custom Commands for Test DRY-ness

**What:** Encapsulate repeated multi-step flows (create event, join event, navigate to event page with mocks) as custom Cypress commands.
**When to use:** Any flow repeated across 2+ spec files.

**Example:**
```javascript
// cypress/support/commands.js
Cypress.Commands.add('mockEventPage', (eventData = {}, playersData = []) => {
  const event = {
    id: 'test-uuid',
    name: 'Test Event',
    status: 'active',
    created_at: '2026-01-01T00:00:00Z',
    ...eventData
  }

  cy.intercept('GET', '**/rest/v1/events*', {
    statusCode: 200,
    body: [event]
  }).as('getEvent')

  cy.intercept('GET', '**/rest/v1/players*', {
    statusCode: 200,
    body: playersData
  }).as('getPlayers')

  // Block Realtime WebSocket to prevent connection errors in mocked tests
  cy.intercept('GET', '**/realtime/**', { statusCode: 200, body: {} }).as('realtimeBlocked')

  cy.visit(`/event/${event.id}`)
  cy.wait('@getEvent')
  cy.wait('@getPlayers')
})

Cypress.Commands.add('getByTestId', (testId) => {
  return cy.get(`[data-testid="${testId}"]`)
})
```

### Pattern 4: Visual Regression at Multiple Breakpoints

**What:** Take full-viewport screenshots at 3 breakpoints and compare against baselines.
**When to use:** Dedicated visual regression spec file.

**Recommended breakpoints (mobile-first app):**

| Name | Width x Height | Rationale |
|------|---------------|-----------|
| mobile | 375 x 812 | iPhone SE / standard mobile (primary target) |
| tablet | 768 x 1024 | iPad portrait |
| desktop | 1280 x 800 | Standard laptop |

**Example:**
```javascript
// cypress/e2e/visual-regression.cy.js
const breakpoints = [
  { name: 'mobile', width: 375, height: 812 },
  { name: 'tablet', width: 768, height: 1024 },
  { name: 'desktop', width: 1280, height: 800 }
]

describe('Visual Regression - Landing Page', () => {
  breakpoints.forEach(({ name, width, height }) => {
    it(`renders correctly at ${name} (${width}x${height})`, () => {
      cy.viewport(width, height)
      cy.visit('/')
      // Wait for fonts to load
      cy.document().its('fonts.status').should('equal', 'loaded')
      cy.compareSnapshot(`landing-${name}`, 0.05)
    })
  })
})

describe('Visual Regression - Event Page', () => {
  breakpoints.forEach(({ name, width, height }) => {
    it(`renders correctly at ${name} (${width}x${height})`, () => {
      cy.viewport(width, height)
      cy.mockEventPage(
        { name: 'Friday Night Commander' },
        [
          { id: 'p1', event_id: 'test-uuid', name: 'Alice', status: 'active', created_at: '2026-01-01T00:01:00Z' },
          { id: 'p2', event_id: 'test-uuid', name: 'Bob', status: 'active', created_at: '2026-01-01T00:02:00Z' },
          { id: 'p3', event_id: 'test-uuid', name: 'Charlie', status: 'dropped', created_at: '2026-01-01T00:03:00Z' }
        ]
      )
      cy.document().its('fonts.status').should('equal', 'loaded')
      cy.compareSnapshot(`event-page-${name}`, 0.05)
    })
  })
})
```

### Pattern 5: Real Supabase Flows (Claude's Discretion)

**What:** Some tests hit the real Supabase backend to validate RLS policies and RPC function behavior.
**Recommendation:** Use real Supabase for these specific flows only:

1. **Event creation via RPC** (`create_event`) -- validates passphrase hashing works, returns UUID
2. **Duplicate player name rejection** -- validates the `UNIQUE(event_id, name)` constraint returns PostgreSQL error code `23505`
3. **Player self-drop via RPC** (`drop_player`) -- validates the RPC correctly changes status and RLS allows the update

Everything else should be mocked because:
- Player list fetching is a simple SELECT -- no complex RLS logic to validate
- Event fetching is a simple SELECT with column-level security (already tested by seeing the right columns)
- QR code rendering is purely frontend
- Visual regression is purely frontend
- Real-time subscriptions are explicitly out of scope

**Test data setup for real flows:**
```javascript
// cypress/support/commands.js
Cypress.Commands.add('createRealEvent', (name, passphrase) => {
  return cy.request({
    method: 'POST',
    url: `${Cypress.env('SUPABASE_URL')}/rest/v1/rpc/create_event`,
    headers: {
      apikey: Cypress.env('SUPABASE_ANON_KEY'),
      'Content-Type': 'application/json',
      Prefer: 'return=representation'
    },
    body: { p_name: name, p_passphrase: passphrase }
  })
})
```

### Anti-Patterns to Avoid
- **`cy.wait(1000)`:** Never use arbitrary timeouts. Use `cy.wait('@alias')` for network requests or `.should()` assertions that auto-retry.
- **CSS class selectors:** Never use `cy.get('.bg-accent')`. Always use `cy.getByTestId('element-name')` or `cy.get('[data-testid="..."]')`.
- **Shared test data across specs:** Each test must create its own data. No global seed file.
- **Testing Realtime in Cypress:** Cypress runs in a single browser; multi-client real-time testing requires tools like Playwright. This is explicitly out of scope.
- **Conditional testing based on DOM state:** Don't use `cy.get('body').then($body => { if ($body.find(...)) ... })`. Design tests with known states.
- **Nesting `.then()` when chaining works:** Use `.should()` chains for assertions, not `.then()` -> `expect()` patterns.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Visual screenshot comparison | Custom pixel-diff logic | `cypress-visual-regression` | Image comparison, threshold math, diff image generation are deceptively complex |
| CI Cypress orchestration | Custom shell scripts for server start/wait | `cypress-io/github-action@v6` with `start` + `wait-on` | Handles npm caching, server readiness detection, timeout, and cleanup automatically |
| Selector helpers | Inline `cy.get('[data-testid="..."]')` everywhere | `cy.getByTestId()` custom command | One place to change the attribute name if convention shifts |
| Supabase API cleanup | Manual DELETE requests in afterEach | API-driven `beforeEach` setup with unique names | Cleanup can fail silently; unique per-test data is isolated by design |
| ESLint Cypress rules | Manual lint config for `cy` globals | `eslint-plugin-cypress` `configs.recommended` | Handles globals declaration, unsafe-to-chain-command, no-async-tests automatically |

**Key insight:** The combination of `cy.intercept()` + fixture files + custom commands eliminates the need for a complex test backend, mock server, or MSW setup. Supabase's PostgREST pattern makes intercept URLs predictable.

## Common Pitfalls

### Pitfall 1: Supabase Realtime WebSocket Errors in Mocked Tests
**What goes wrong:** When tests mock PostgREST endpoints but don't handle Realtime, Cypress console fills with WebSocket connection errors. The app's `useEventChannel` hook tries to open a Realtime subscription.
**Why it happens:** The Supabase client always attempts Realtime connections when components mount.
**How to avoid:** Intercept and stub the Realtime WebSocket endpoint in mocked tests: `cy.intercept('GET', '**/realtime/**', { statusCode: 200, body: {} })`. Alternatively, block it entirely.
**Warning signs:** Red WebSocket errors in browser console during test runs; flaky assertion timeouts.

### Pitfall 2: localStorage/sessionStorage State Leaking Between Tests
**What goes wrong:** A player identity stored in localStorage from a previous test causes the next test to skip the join form.
**Why it happens:** Cypress does not clear localStorage/sessionStorage between tests by default.
**How to avoid:** Call `cy.clearLocalStorage()` and `cy.window().then(win => win.sessionStorage.clear())` in `beforeEach`, or use `cy.session()` intentionally.
**Warning signs:** Test passes in isolation but fails when run in suite; join form not visible when expected.

### Pitfall 3: Font Loading Race in Visual Regression
**What goes wrong:** Visual regression screenshots capture the page before Google Fonts (Cinzel, Inter) finish loading, resulting in fallback font diffs.
**Why it happens:** The app loads fonts from Google Fonts CDN with `font-display: swap`. The swap period shows fallback fonts briefly.
**How to avoid:** Wait for fonts to load before taking screenshots: `cy.document().its('fonts.status').should('equal', 'loaded')`.
**Warning signs:** Visual regression failures showing font differences only; baselines look different on first vs subsequent runs.

### Pitfall 4: Vite Dev Server Not Ready in CI
**What goes wrong:** Cypress starts before the Vite dev server is serving, causing all `cy.visit()` calls to fail.
**Why it happens:** GitHub Actions starts the server and tests concurrently without waiting for HTTP readiness.
**How to avoid:** Use `cypress-io/github-action@v6` with `start: npm run dev -- --host` and `wait-on: 'http://localhost:5173'`. The `--host` flag is required for the wait-on URL to work in Linux CI.
**Warning signs:** First test fails with "page not found" or connection refused; subsequent tests may pass if server starts in time.

### Pitfall 5: ESM / CJS Config Mismatch
**What goes wrong:** Cypress config file fails to load with "require is not defined in ES module scope" or "Cannot use import statement outside a module".
**Why it happens:** The project has `"type": "module"` in package.json, which makes all `.js` files ESM by default. Cypress config needs to match.
**How to avoid:** Use `cypress.config.js` with ESM `import`/`export` syntax (the project's `"type": "module"` in package.json makes this work). All Cypress support/command files should also use ESM imports.
**Warning signs:** Cypress fails to open or run with module-related errors.

### Pitfall 6: Visual Regression Baselines Not Committed
**What goes wrong:** CI fails on visual regression because baseline images don't exist in the repository.
**Why it happens:** Baselines are generated locally but `.gitignore` excludes them, or they were never committed.
**How to avoid:** Commit `cypress/snapshots/base/` to the repo. Add `cypress/snapshots/actual/` and `cypress/snapshots/diff/` to `.gitignore`. Document the baseline update workflow.
**Warning signs:** All visual regression tests fail in CI with "baseline not found" errors.

### Pitfall 7: IPv6 localhost Issue in GitHub Actions
**What goes wrong:** `wait-on` hangs or times out trying to reach `localhost:5173` in CI.
**Why it happens:** Ubuntu runners in GitHub Actions resolve `localhost` to `::1` (IPv6) but Vite may only listen on `127.0.0.1`.
**How to avoid:** Use `http://127.0.0.1:5173` instead of `http://localhost:5173` in the `wait-on` URL, or use `--host` flag with Vite which binds to all interfaces.
**Warning signs:** CI workflow times out at the "wait for server" step.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `cy.route()` (Cypress <6) | `cy.intercept()` | Cypress 6.0+ (2020) | Full HTTP layer interception, not just XHR |
| `cypress/integration/` folder | `cypress/e2e/` folder | Cypress 10+ (2022) | Default spec location changed |
| `cypress.json` config | `cypress.config.js` with `defineConfig()` | Cypress 10+ (2022) | JS config with IDE autocomplete |
| `cypress/support/index.js` | `cypress/support/e2e.js` | Cypress 10+ (2022) | Separate support files for e2e vs component |
| `.eslintrc` with eslint-plugin-cypress/recommended | Flat config with `pluginCypress.configs.recommended` | eslint-plugin-cypress 5.0+ (2025) | Legacy config deprecated; `/flat` suffix removed |
| `cypress run --record` (free) | Cypress Cloud (paid) | 2023+ | Recording to Cypress Cloud now requires paid plan; local videos/screenshots are free |

**Deprecated/outdated:**
- `cy.server()` / `cy.route()`: Replaced by `cy.intercept()` in Cypress 6+
- `cypress.json`: Replaced by `cypress.config.{js,ts,mjs}` in Cypress 10+
- `cypress/integration/`: Renamed to `cypress/e2e/` in Cypress 10+
- `eslint-plugin-cypress/flat`: The `/flat` suffix was deprecated in v5.0.0; import directly from `eslint-plugin-cypress`

## Open Questions

1. **Supabase Environment for CI**
   - What we know: Real Supabase tests need valid `SUPABASE_URL` and `SUPABASE_ANON_KEY` in CI
   - What's unclear: Whether to use the same Supabase project as development or a dedicated test project
   - Recommendation: Use the same project for now (Phase 1 has no destructive operations); if test isolation becomes a concern, create a separate Supabase project later. Store credentials as GitHub Actions secrets.

2. **Visual Regression Baseline Generation Workflow**
   - What we know: Baselines must be generated locally and committed before CI can run regression checks
   - What's unclear: How to handle baseline updates when UI intentionally changes
   - Recommendation: Add npm scripts: `"cy:base"` to generate baselines (`CYPRESS_visualRegressionType=base npx cypress run`) and `"cy:test"` for regression runs. Document that developers must run `cy:base` and commit updated baselines after intentional UI changes.

3. **Realtime WebSocket Blocking Strategy**
   - What we know: Mocked tests will produce WebSocket errors from Realtime subscriptions
   - What's unclear: Whether intercepting `**/realtime/**` is sufficient or if specific WebSocket interception is needed
   - Recommendation: Start with `cy.intercept('GET', '**/realtime/**', ...)` for the HTTP upgrade request. If WebSocket errors persist, consider using `Cypress.on('uncaught:exception')` to suppress Realtime-specific errors in mocked tests.
