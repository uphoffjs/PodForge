# Phase 1.1: Cypress E2E Test Infrastructure and Phase 1 Flow Tests - Context

**Gathered:** 2026-02-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Set up Cypress E2E testing infrastructure and write comprehensive tests covering all Phase 1 user flows: event creation, player join, QR code, self-drop, duplicate name rejection, and the mobile-first dark theme. This phase does NOT add new features — it tests what Phase 1 already built.

</domain>

<decisions>
## Implementation Decisions

### Test Flow Coverage
- Comprehensive coverage: happy paths, error cases, and edge cases (empty states, boundary conditions)
- Skip real-time multi-client simulation in E2E — real-time subscription behavior is out of scope for Cypress tests
- QR code: verify element is present and encoded URL matches the event link (no visual snapshot of QR itself)
- Full visual regression testing at multiple breakpoints for the mobile-first dark theme

### Supabase Test Strategy
- Most tests use `cy.intercept()` to mock Supabase API responses for reliability and speed
- Some flows hit real Supabase where RLS/RPC behavior must be verified (Claude's discretion on which flows)
- Test data for real Supabase flows: API-driven setup in `beforeEach` — each test creates its own data, no shared seed/teardown dependency
- Fixture data: mix of static JSON fixtures (common response shapes) and dynamic test helpers (unique event names, player names)

### Test Infrastructure
- Custom Cypress commands (e.g., `cy.createEvent()`, `cy.joinPlayer()`) — no Page Object pattern
- Test specs organized by user flow: `event-creation.cy.js`, `player-join.cy.js`, `self-drop.cy.js`, etc.
- JavaScript for all Cypress files (not TypeScript)
- `data-testid` attributes on React components for selectors (never CSS classes or tag names)

### CI & Failure Handling
- GitHub Actions: run Cypress on every PR and push to main
- Capture screenshots AND video on test failure, uploaded as GitHub Actions artifacts
- No retries — tests must be deterministic; fail on first attempt everywhere
- Required status check: Cypress must pass before PR can merge

### Claude's Discretion
- `data-testid` naming convention (e.g., kebab-case with hierarchy vs flat)
- Which specific flows hit real Supabase vs mocked (based on risk analysis of RLS/RPC behavior)
- Visual regression tooling choice and threshold configuration
- Exact custom command API design

</decisions>

<specifics>
## Specific Ideas

- Follow Cypress best practices: use `cy.intercept()` for API mocking/waiting, `.should()` chains for auto-retry assertions, never `cy.wait(ms)` with arbitrary timeouts
- Install `eslint-plugin-cypress` and scope ESLint config to test files
- The concern about seed/teardown is reliability: if cleanup fails, stale data shouldn't break future tests — hence API-driven setup per test

</specifics>
