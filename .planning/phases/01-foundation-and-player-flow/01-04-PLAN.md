  <name>Task 1: Create Realtime subscription and visibility refetch hooks</name>
  <files>
    src/hooks/useEventChannel.ts
    src/hooks/useVisibilityRefetch.ts

  <action>
    **src/hooks/useEventChannel.ts** -- Supabase Realtime subscription:
    - `useEventChannel(eventId: string)` hook
    - Creates a single Realtime channel per event: `supabase.channel(`event:${eventId}`)`
    - Subscribes to `postgres_changes` for two tables on this channel:
      1. `players` table filtered by `event_id=eq.${eventId}` -- event: `*` (INSERT, UPDATE, DELETE)
      2. `events` table filtered by `id=eq.${eventId}` -- event: `*`
    - On any change from players: `queryClient.invalidateQueries({ queryKey: ['players', eventId] })`
    - On any change from events: `queryClient.invalidateQueries({ queryKey: ['event', eventId] })`
    - **CRITICAL cleanup:** Return `() => { supabase.removeChannel(channel) }` from useEffect. Use `removeChannel` not just `unsubscribe` -- full cleanup prevents React Strict Mode double-subscription bug (see RESEARCH.md Pitfall 5).
    - Dependencies: `[eventId, queryClient]`
    - Do NOT create separate channels per table. One channel with multiple `.on()` handlers is correct (see RESEARCH.md anti-patterns).

    **src/hooks/useVisibilityRefetch.ts** -- Tab restore handler:
    - `useVisibilityRefetch(eventId: string)` hook
    - Listens to `document.visibilitychange` event
    - When `document.visibilityState === 'visible'`: invalidate both `['players', eventId]` and `['event', eventId]` queries
    - This catches stale state after mobile users background the app and return
    - Proper cleanup: remove event listener on unmount
    - Dependencies: `[eventId, queryClient]`

    **Anti-pattern reminder from RESEARCH.md:** Do NOT directly mutate React state from Realtime payloads. Always use `invalidateQueries` to trigger a fresh fetch. DELETE events with RLS only deliver primary keys, not full records, making direct state merging unreliable.
  </action>
  <verify>
    `npx tsc --noEmit` passes.
    Both hooks export correctly and can be imported.
  </verify>
  <done>
    useEventChannel subscribes to Realtime postgres_changes on players and events tables for a specific event, invalidating React Query caches on changes. useVisibilityRefetch invalidates caches when the tab becomes visible. Both have proper cleanup.
  </done>

    - `onError`: toast.error("Failed to leave event. Please try again.")
    - After dropping, the EventPage should show the join form again (since stored player ID is cleared)

    - Props: `isOpen: boolean`, `title: string`, `message: string`, `confirmLabel: string`, `onConfirm: () => void`, `onCancel: () => void`, `isLoading?: boolean`
    - Dark overlay + centered card (same styling pattern as CreateEventModal)
    - Two buttons: Cancel (secondary, text-text-secondary) and Confirm (destructive, bg-error)
    - Confirm button shows loading state when isLoading is true
    - This component will be reused for other confirmations in later phases

    **src/pages/EventPage.tsx** -- Wire everything together:

    - Add `useVisibilityRefetch(eventId)` call to enable tab restore
    - Add "Leave Event" button:

      - Place the button below the player list section, not inline with player names
      - Styled as a subtle danger action: outline button with `text-error border-error` (not a big red button -- it should be findable but not accidentally tappable)
      - Only visible when the current user is an active player (has stored player ID that's in the active list)
    - On "Leave Event" click:
      1. Per user decision: "Confirmation required before dropping"
      2. Open ConfirmDialog with: title "Leave Event?", message "You'll be marked as dropped. You can ask the admin to re-add you later.", confirmLabel "Leave Event"
      3. On confirm: call `useDropPlayer` mutation with the stored player ID
      4. On success: dialog closes, player ID cleared from localStorage, EventPage re-renders showing join form
