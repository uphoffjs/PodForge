**Fix 2 — Fix validation effect race condition (Finding 4, part 2):**

The validation effect in `EventPage.tsx` (lines 53-61) runs every time `players` changes. After a successful join:
1. `handleJoined` sets `currentPlayerId` and calls `storePlayerId`
2. React Query invalidates players query
3. The players refetch begins (but `players` is still the OLD array without the new player)
4. The validation effect fires, sees `currentPlayerId` is set, but the old `players` array doesn't include it
5. Effect clears the identity — join form flashes back

Fix: Add a `joinInFlight` ref that tracks whether a join is in progress. Set it to true when the join form triggers `handleJoined`, and clear it after the next successful players refetch that includes the player.

Revised approach — simpler: The validation effect should skip clearing when `currentPlayerId` was JUST set in the same render cycle. Use a `justJoinedRef` to gate the validation effect:

```tsx
const justJoinedRef = useRef(false)

const handleJoined = useCallback(
  (playerId: string) => {
    if (eventId) {
      storePlayerId(eventId, playerId)
    }
    justJoinedRef.current = true
    setCurrentPlayerId(playerId)
  },
  [eventId],
)

// Verify stored player exists in fetched player list
useEffect(() => {
  if (!eventId || !players || !currentPlayerId) return

  // Skip validation right after joining — players list hasn't refetched yet
  if (justJoinedRef.current) {
    const playerExists = players.some((p) => p.id === currentPlayerId)
    if (playerExists) {
      // Player appeared in the list — join is confirmed, resume normal validation
      justJoinedRef.current = false
    }
    // Either way, don't clear identity while join is pending
    return
  }

  const playerExists = players.some((p) => p.id === currentPlayerId)
  if (!playerExists) {
    clearPlayerId(eventId)
    setCurrentPlayerId(null)
  }
}, [eventId, players, currentPlayerId])
```

This way:
- Normal validation works as before for returning visitors whose player was dropped
- After a fresh join, the effect defers clearing until the players list actually includes (or permanently excludes) the player
- Once the player appears in the refetched list, `justJoinedRef` resets and normal validation resumes

The validation effect calls `setCurrentPlayerId(null)` — this is correct reactive behavior (clearing stale state). To satisfy the lint rule, wrap the state-setting logic in a callback pattern or refactor as needed. Check what the specific ESLint rule expects.

Actually, `react-hooks/set-state-in-effect` is an experimental/non-standard rule. Run `npx eslint src/pages/EventPage.tsx` to see the exact errors and their locations. Then apply the minimum fix:

- Most likely fix: The new-player-detection effect (lines 64-87) calls `setNewPlayerIds(added)` and `setNewPlayerIds(new Set())` — these should use functional updates. The validation effect calls `setCurrentPlayerId(null)`.
