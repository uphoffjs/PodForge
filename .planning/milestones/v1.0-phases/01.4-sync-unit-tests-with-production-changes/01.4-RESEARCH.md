## Summary

Phase 1.2 made two production code changes that invalidated unit tests: (1) `storePlayerId` was moved out of the `useJoinEvent` hook's `onSuccess` callback into `EventPage.handleJoined`, and (2) the single `data-testid="join-error"` element in `JoinEventForm` was split into two distinct testids: `join-validation-error` (for client-side validation) and `join-mutation-error` (for server/mutation errors). These changes are correct at the production level -- 44/44 E2E tests pass -- but 8 unit tests were never updated to match.

This is a pure test-maintenance phase. No production code changes are needed. The scope is narrow, well-defined, and fully verified by running the test suite.

**Primary recommendation:** Update the 8 failing unit test assertions to match the current production testids and hook behavior. Remove the stale `storePlayerId` assertion from `useJoinEvent.test.ts` (the hook no longer calls it), and replace all `join-error` testid queries in `JoinEventForm.test.tsx` with the correct `join-validation-error` or `join-mutation-error` testid based on which error path each test exercises.

### Pattern 1: Hook Test with vi.mock (existing pattern)
**What:** Tests mock Supabase client and utility functions via `vi.mock`, then verify hook behavior
**When to use:** Testing React Query mutation hooks in isolation
**Example from codebase:**

### Pattern 2: Component Test with Mocked Hook (existing pattern)
**What:** Tests mock the entire hook module, control the returned state object, then assert rendered output
**When to use:** Testing component rendering and interaction without actual mutation logic
**Example from codebase:**
```typescript
// src/components/JoinEventForm.test.tsx (current pattern)
vi.mock('@/hooks/useJoinEvent', () => ({
  useJoinEvent: () => hookState,
}))
// hookState is mutable -- tests set isPending, isError, error before rendering
```

### Pattern 3: TestId-Based Assertions (project convention)
**What:** All DOM queries use `data-testid` attributes, queried via `screen.getByTestId()` / `screen.queryByTestId()`
**When to use:** All component tests -- project convention from Phase 01.1 decision
**Key convention:** Hierarchical kebab-case: `join-form`, `join-name-input`, `join-validation-error`, `join-mutation-error`

### Anti-Patterns to Avoid
- **Blind find-and-replace of testids:** Some tests check for validation errors (client-side), others check for mutation errors (server-side). Using the wrong replacement testid would make tests pass but test the wrong thing.
- **Removing the storePlayerId test entirely:** The behavior still exists, it just moved to EventPage. The unit test for the hook should be updated to verify the hook does NOT call storePlayerId (since that is now the caller's responsibility).

## Common Pitfalls

### Pitfall 1: Mapping Tests to Wrong Error Testid
**What goes wrong:** Replacing all `join-error` references with `join-validation-error` when some tests actually exercise the mutation error path (server errors, duplicate names).
**Why it happens:** Both validation and mutation errors previously shared one testid. After the split, each test must be classified by which error path it triggers.
**How to avoid:** Categorize each failing test by its trigger:
- Tests that submit invalid input (empty, short names) -> `join-validation-error`
- Tests that set `hookState.isError = true` with error codes -> `join-mutation-error`
- Tests that clear validation error on resubmit -> `join-validation-error` (for the initial check) + `queryByTestId` null check still uses `join-validation-error`
**Warning signs:** A test passing but asserting against the wrong testid (e.g., validation error test checking mutation error element).

### Pitfall 2: Forgetting queryByTestId Negative Assertions
**What goes wrong:** Only updating `getByTestId('join-error')` calls but missing `queryByTestId('join-error')` calls that assert the error is NOT present.
**Why it happens:** `queryByTestId` returns null instead of throwing, so it's used for absence checks. Easy to miss in a search.
**How to avoid:** Search for ALL occurrences of `join-error` in the test file -- both `getByTestId` and `queryByTestId`.
**Warning signs:** Tests pass but negative assertions are checking for absence of the wrong testid.

### Pitfall 3: Removing storePlayerId Mock Without Verifying No Other Test Needs It
**What goes wrong:** Removing the `vi.mock('@/lib/player-identity')` entirely, breaking other tests that might reference it.
**Why it happens:** The failing test is the only one that asserts storePlayerId was called, but the mock may affect other test behavior.
**How to avoid:** The mock can stay in place (it won't hurt); just update the one test that asserts `mockStorePlayerId.toHaveBeenCalledWith(...)`.
**Warning signs:** Other tests in the same file start failing after removing the mock.

## Code Examples

### Fix 1: useJoinEvent.test.ts -- Remove Stale storePlayerId Assertion

The `useJoinEvent` hook no longer calls `storePlayerId` directly. Phase 1.2 moved that responsibility to `EventPage.handleJoined`:

**Production code (current):**
```typescript
// src/hooks/useJoinEvent.ts -- onSuccess does NOT call storePlayerId
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ['players', eventId] })
  toast.success('Joined the event!')
},
```

```typescript
// src/pages/EventPage.tsx -- handleJoined calls storePlayerId now
const handleJoined = useCallback(
  (playerId: string) => {
    if (eventId) {
      storePlayerId(eventId, playerId)  // <-- moved here in Phase 1.2
    }
    justJoinedRef.current = true
    setCurrentPlayerId(playerId)
  },
  [eventId],
)
```

**Failing test (line 78-91):**
```typescript
it('calls storePlayerId with eventId and player.id on success', async () => {
  // ... setup and mutate ...
  expect(mockStorePlayerId).toHaveBeenCalledWith('evt1', 'player-uuid-123')
  // ^^^ This assertion fails because useJoinEvent no longer calls storePlayerId
})
```

**Fix approach:** Update the test to verify that `storePlayerId` is NOT called by the hook (since the hook no longer has that responsibility). Alternatively, the test could be renamed and restructured to verify the hook returns data correctly (which it does -- the caller uses that data to call storePlayerId).
